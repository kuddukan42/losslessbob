# Master Data vs. User Data

LosslessBob's SQLite database contains two distinct categories of data. Understanding the split matters when installing curator updates, exporting a snapshot, or debugging what survives a master import.

---

## Master Data

Shared reference data curated and published by the project maintainer. All users receive the same master dataset when they install a Master Update. This data describes the LosslessBob archive itself — not anything specific to a local collection.

**Tables** (`MASTER_TABLES` in `backend/db.py`):

| Table | Contents |
|---|---|
| `checksums` | All known file checksums from the archive |
| `entries` | LB entry metadata (title, date, venue, etc.) |
| `entry_files` | Individual file records per entry |
| `entry_changes` | Tracked changes to entries over time |
| `lb_master` | Status classification for every LB number (public/private/missing/nonexistent) |
| `lb_status_history` | Audit log of lb_master status transitions |
| `lb_missing` | LB numbers confirmed never to have had a page |
| `lb_alias` | Disambiguation aliases for LB numbers |
| `lb_problems` | Known quality problems per LB (corrupt files, mislabelled, etc.) |
| `flat_file_releases` | Catalog of flat-file update releases |
| `flat_file_changelog` | Per-release change records |
| `bootleg_titles` | LBBCD bootleg-CD catalog |
| `bootleg_scrapes` | Raw scrape data backing the bootleg catalog |
| `location_geocoded` | Lat/lon results for concert locations |
| `dylan_performances` | Dylan setlist / performance reference data |

> `entries_fts` is a virtual FTS5 table mirrored from `entries` via triggers. It is rebuilt automatically after a master import and is not copied directly.

**`meta` keys that ship with master** (`MASTER_META_KEYS`):

- `import_hash`, `last_import_date`, `last_lb_number`
- `master_version`, `master_published_at`, `master_schema_version`

---

## User Data

Data generated by a specific user's local activity. Never exported or overwritten by a master update. Survives intact when a master snapshot is installed.

**Tables** (`USER_TABLES` in `backend/db.py`):

| Table | Contents |
|---|---|
| `my_collection` | Folders the user has linked to LB entries |
| `collection_meta` | Per-folder metadata (tags, notes) |
| `my_wishlist` | LB entries the user wants to acquire |
| `integrity_events` | Results of local collection integrity scans |
| `torrents` | Torrent download records |
| `rename_history` | Log of folder rename operations |
| `forum_posts` | Cached WTRF forum posts |
| `folder_lb_link` | Sticky folder→LB mappings |
| `scrape_sessions` | Per-session scrape run records |
| `site_inventory` | Local snapshot of the LosslessBob site index |

**`meta` keys that are user-local** (`USER_META_KEYS`):

- Scraper settings: `auto_scrape`, `scrape_delay_ms`, `scrape_attachments`, `force_scrape`, `download_files`, `use_local_pages`
- UI settings: `search_page_size`
- qBittorrent settings: `qbt_host`, `qbt_port`, `qbt_category`, `qbt_tags`, `tracker_list`
- Forum credentials: `wtrf_board_id`, `wtrf_username`, `wtrf_password`
- `is_curator`

User `meta` keys are **never** included in a master export. The export uses `MASTER_META_KEYS` as a whitelist and the verifier asserts that no user tables or user meta leaked into the snapshot.

Other user-local state outside the database:
- `data/gui_state.json` — column widths, window geometry

---

## How the Split Is Enforced

**Export** (`POST /api/master/export`, curator-only):

1. `VACUUM INTO` a fresh snapshot file.
2. Drop every `USER_TABLES` table from the snapshot.
3. Delete every `meta` row whose key is not in `MASTER_META_KEYS`.
4. Stamp `master_version`, `master_published_at`, `master_schema_version`.
5. Verify: assert no `USER_TABLES` tables remain and no non-master meta keys are present.
6. Write SHA256 + `.manifest.json` sidecar.

**Import** (`POST /api/master/import`):

1. Validate manifest SHA256 — reject on mismatch.
2. Refuse snapshots with `master_schema_version` newer than `MASTER_SCHEMA_VERSION` (the app would not understand the schema).
3. Take a `pre_master_import` backup of the live database.
4. `ATTACH` the snapshot, copy only `MASTER_TABLES` rows into the live DB (replacing existing rows).
5. Replace only `MASTER_META_KEYS` rows in `meta`; all user meta keys are untouched.
6. Rebuild `entries_fts`.

User data is never read from the snapshot — only the tables explicitly listed in `MASTER_TABLES` are touched.

---

## Schema Versioning

`MASTER_SCHEMA_VERSION` (integer, defined in `backend/db.py`) is bumped whenever a structural change is made to master tables. The import guard prevents an older client from installing a snapshot it cannot interpret. The current version and its history of bumps are documented inline in `db.py`.
