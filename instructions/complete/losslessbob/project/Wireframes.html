01 Architecture

Information architecture

Your current top bar is a flat row of 14 tabs in roughly the order features were added. Below, the same 14 screens are grouped by what the user is trying to do, so the nav reads top-to-bottom as a sequence of jobs rather than a feature list. A left sidebar with collapsible groups replaces the tab row.

Settings

One screen for app config, with theme controls promoted to a header button (open anywhere).

02Shell

App shell

A persistent left sidebar (grouped, collapsible), a slim top bar holding breadcrumb + global actions (theme, DB indicator), and a status footer that survives across every screen. Sub-tabs (for things like My Collection's Missing / Wishlist / Duplicates) live inside the content area, not in the global nav.

08Redesign Notes

Recurring patterns to lock in

Three decisions show up in every screen above. Calling them out separately so we can debate them once and apply consistently rather than re-deciding per-tab.

A · Status legibility in tables

Today, whole rows are tinted green / red / yellow / pink for status. This wrecks legibility under any theme and forces theming to stay close to the current pastel palette — which is exactly what you flagged.

Proposed

1. A thin left-edge color bar (3 px) carries the row state.
2. A status pill in a dedicated column carries the label.
3. Row background stays neutral, alternating only for striping. Whole-row tint is reserved for selection.

LB-810 · listbox · 14/14 · Matched
LB-1964 · listbox · 2/9 · Incomplete
bd2025-07-25 · 0/48 · Not Found

B · Theming = mode × accent, nothing more

Today's Themes screen has ~20 individually-editable colors. Custom users will keep that option, but the default surface is just Light / Dark / System × one accent. Status colors (green / amber / red) are fixed across themes for accessibility and shifted to the left-edge bar + pill so they never collide with the accent.

Why this is safer

Accent only paints: primary buttons, active sidebar item, focused input outline, selection highlight, and progress fills. It never paints a table row. That's how a Plum or Rust accent stays legible.

C · Density & column widths

Every screen here uses uniform table styling. Three density presets — Comfortable / Default / Compact — drive row height and padding globally. Per-screen column widths still persist via "Save column widths" in Setup → Preferences, which already exists today.

D · Cross-screen actions on any LB row

Every table row showing an LB# gets the same right-click menu: Open in Search, Open in Bootlegs, Open Attachments, Open Spectrograms, Reveal on disk, Copy LB#. This means you don't have to choose the "right tab" first — any list is a launchpad to any other view.
